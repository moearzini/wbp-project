Bausteine: Manifest, application.js & App Registry

Vertiefung · Technische Architektur

Um die CMS-Idee umzusetzen, wurde das System bewusst in klar getrennte Bausteine aufgeteilt. Jeder Baustein übernimmt eine klar definierte Rolle – ähnlich wie in echten Content-Management-Systemen.

1) App Registry – der Einstiegspunkt

Die App Registry ist der zentrale Überblick über alle verfügbaren Apps (z. B. Lectures, Tools, Spotlight).

  • legt fest, welche Apps existieren
  • verweist auf den jeweiligen App-Pfad
  • entspricht konzeptionell einem App-Menü oder Modulverzeichnis

2) application.js – das Manifest der App

Für jede App existiert eine eigene application.js. Diese Datei fungiert als Manifest der App.

  • definiert Sections (z. B. Vorlesungen)
  • definiert Pages innerhalb einer Section
  • verknüpft Pages mit Content über src

In einem klassischen CMS würde diese Information typischerweise in einer Datenbank liegen – hier ist sie bewusst als strukturierte JavaScript-Logik umgesetzt.

Beispiel: Manifest-Struktur in application.js

Das folgende Beispiel zeigt vereinfacht, wie Sections und Pages im Manifest definiert werden. Diese Struktur ersetzt im Prinzip eine CMS-Datenbank. 

// application.js (vereinfacht)
lectures.config = {
  sections: [
    {
      id: "vl1",
      title: "Vorlesung 1 – Einführung",
      pages: [
        {
          id: "vl1-ueberblick",
          title: "Überblick & Ziel der Vorlesung",
          src: "content/vl1/ueberblick.html"
        }
      ]
    }
  ]
};

3) HTML-Seiten – reiner Content

Der eigentliche Inhalt liegt vollständig getrennt in eigenen HTML-Dateien:

  • keine Navigation
  • keine App-Logik
  • nur Text, Struktur und ggf. Grafiken

Damit entspricht jede HTML-Datei inhaltlich einer klassischen CMS-Seite.

Beispiel: Registrierung der App

Die App Registry stellt sicher, dass die Lectures-App im System bekannt ist und geladen werden kann. 

// index.html / app bootstrap
window.APP_REGISTRY = {
  lectures: {
    id: "lectures",
    title: "WebProg Lectures",
    path: "resources/lectures"
  }
};

Kurzer Ausblick: Technische Herausforderungen

  • Laden von HTML-Dateien ohne Webserver (file://)
  • Trennung von App-Titel und Seiten-Titel
  • Dynamisches Rendering ohne Framework

Diese Punkte werden auf der nächsten Vertiefungsseite gezielt aufgegriffen.

Merksatz für die Präsentation:
Die App Registry sagt was es gibt, das Manifest (application.js) beschreibt wie es aufgebaut ist, und die HTML-Seiten enthalten was angezeigt wird.
